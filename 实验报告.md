## 一、实验概述

### 1.1 实验目标

本实验要求学生在掌握智能体基本构建方法的基础上，结合 Docker 容器化、Kubernetes 集群编排、Prometheus 监控采集、Jaeger 分布式追踪等技术，自主设计并实现一个具备实际运维价值的智能体应用。

```python
@tool("file_read_tool")
def file_read_tool(file_path: str) -> str:
    """
    Read contents of a file.

    Args:
        file_path: the path of this file
    """
    try:
        with open(file_path, "r", encoding="utf-8") as f:
            content = f.read()
        return content
    except FileNotFoundError:
        return f"{file_path} doesn't exist"


@tool("file_write_tool")
def file_write_tool(content: str, file_path: str) -> str:
    """
    Write content into a file at the specified path.

    Args:
        content: The content to write into the file.
        file_path: The full path of the file to be written.
    """
    import os
    os.makedirs(os.path.dirname(file_path), exist_ok=True)
    with open(file_path, "w", encoding="utf-8") as f:
        f.write(content)
    return f"Successfully wrote content to {file_path}"
```

> **说明**：与手册示例一致，封装了 Python 文件读写接口，工具函数带有完整的注释（docstring），LangChain 会从中提取工具描述。

```python
@tool("container_exec_cmd_tool")
def container_exec_cmd_tool(container_name: str, command: str) -> str:
    """Execute a shell command inside an already running Docker container.

    Args:
        container_name: The name of the container
        command: The command to execute
    """
    import docker
    client = docker.from_env()
    container = client.containers.get(container_name)
    exit_code, output = container.exec_run(cmd=command, stdout=True, stderr=True)
    output_text = output.decode(errors="ignore")
    return f"Exit code: {exit_code}\n\nOutput:\n{output_text}"
```

> **说明**：通过对 docker SDK 的 `exec_run` 函数封装实现，与手册一致。

### 2.3 智能体构建（对应手册 4.6 节）

**文件：`core/agent_builder.py`**

```python
from langchain.agents import initialize_agent, AgentType

class OpsAgent:
    def __init__(self, logger=None):
        self.llm = init_llm()
        self.tools = [
            dockerfile_generate_tool,
            file_read_tool, file_write_tool,
            image_build_tool, container_run_tool,
            container_exec_cmd_tool, failure_diagnosis_tool,
        ]
        self.agent = initialize_agent(
            tools=self.tools,
            llm=self.llm,
            agent=AgentType.STRUCTURED_CHAT_ZERO_SHOT_REACT_DESCRIPTION,
            verbose=True,
            max_iterations=30,
        )

    def invoke(self, query):
        response = self.agent.invoke({"input": query})
        print(f"Final Answer: {response}")
```

> **说明**：与手册示例一致，使用 `initialize_agent` 创建 ReAct 智能体，指定 `STRUCTURED_CHAT_ZERO_SHOT_REACT_DESCRIPTION` 类型，代表该智能体是结构化输出、零样本提示、ReAct 行为模式。

## 四、智能监控问答助手（场景一）

### 4.1 场景描述

在实验二中，我们通过 Python 脚本调用 Prometheus API，手动编写 PromQL 查询语句来获取集群的 CPU 占用率、内存使用量、Pod 重启次数等指标。这种方式要求运维人员具备 PromQL 语法知识，使用门槛较高。

本场景将 Prometheus 查询能力封装为**具有清晰语义的智能体工具集**，使运维人员可以通过**自然语言**向智能体提问，由智能体自主选择合适的 PromQL 查询语句、调用 Prometheus API、汇总数据并生成可读的分析报告。进阶实现还结合了 matplotlib 可视化库，让智能体在回答数据趋势问题时自动生成折线图。

### 4.2 设计思路

本场景的核心设计哲学是：**"工具即语义"**——将 Prometheus 的查询接口封装为多个具有清晰语义的原子工具，每个工具：

1. **名称自解释**：如 `query_pod_cpu_usage`、`query_pod_restart_count`，LLM 可直接理解工具用途
2. **描述引导**：精心编写的工具描述（Description）包含典型提问示例，引导 LLM 在不同问题下选择正确的工具组合
3. **参数友好**：参数使用 `namespace`、`pod_name` 等自然语言概念，而非底层 PromQL 语法
4. **结果结构化**：返回格式化文本，包含数据、汇总统计和智能分析结论

工具之间形成**层次化体系**：
- **底层工具**：`prometheus_query_tool` —— 执行原始 PromQL，供高级用户或内部调用
- **领域工具**：`query_pod_cpu_usage`、`query_pod_memory_usage` 等 —— 封装特定指标的查询逻辑
- **聚合工具**：`query_namespace_health_report` —— 组合多个领域工具生成一站式报告
- **趋势工具**：`query_cpu_trend`、`query_memory_trend` —— 处理时序数据并支持可视化

### 4.3 实现的监控工具集

本场景在原有 `prometheus_query_tool` 基础上，新增并重构了 **10 个**监控专用工具，涵盖 CPU、内存、重启、网络、健康总览、趋势分析和可视化等能力：

#### 4.3.1 底层通用查询工具

**工具名称**：`prometheus_query_tool`

| 属性 | 内容 |
|------|------|
| 功能 | 执行原始 PromQL 查询，支持即时查询和范围查询两种模式 |
| 内核 | 调用 `prometheus_api_client` 的 `custom_query` / `custom_query_range` |
| 用途 | 当专用工具无法覆盖的高级查询场景时使用 |

#### 4.3.2 CPU 监控工具

**工具名称**：`query_pod_cpu_usage`

| 属性 | 内容 |
|------|------|
| 典型提问 | "查看 bookinfo 各服务的 CPU 使用情况"、"productpage 的 CPU 占用" |
| PromQL 模板 | `sum(rate(container_cpu_usage_seconds_total{namespace="..."}[5m])) by (pod)` |
| 输出特性 | 按 CPU 使用量降序排列，自动格式化（cores/m），计算占比，标注异常 |

**工具名称**：`query_cpu_trend`

| 属性 | 内容 |
|------|------|
| 典型提问 | "reviews-v2 最近五分钟的 CPU 使用趋势"、"bookinfo CPU 趋势图" |
| 查询类型 | 范围查询，返回时序数据点 |
| 可视化 | `save_chart=True` 时自动调用 matplotlib 生成折线图并保存为 PNG |
| 趋势分析 | 计算前半段/后半段均值变化，判断"显著上升/下降/平稳" |

#### 4.3.3 内存监控工具

**工具名称**：`query_pod_memory_usage`

| 属性 | 内容 |
|------|------|
| 典型提问 | "bookinfo 各服务的内存占用"、"reviews-v2 内存用了多少" |
| PromQL 模板 | `sum(container_memory_working_set_bytes{namespace="...",container!=""}) by (pod)` |
| 输出特性 | 自动转换为 MiB 格式，检测超过 1 GiB 的高内存预警 |

**工具名称**：`query_memory_trend`

| 属性 | 内容 |
|------|------|
| 典型提问 | "productpage 内存变化趋势"、"bookinfo 各服务的内存趋势图" |
| 可视化 | 自动生成折线图，纵轴单位为 MiB |
| 趋势分析 | 识别持续增长趋势（可能存在内存泄漏风险） |

> **设计要点**：使用 `container_memory_working_set_bytes` 而非 `container_memory_usage_bytes`，因为 Working Set 更能反映容器实际使用的物理内存，排除缓存影响。

#### 4.3.4 重启与稳定性监控

**工具名称**：`query_pod_restart_count`

| 属性 | 内容 |
|------|------|
| 典型提问 | "查看 bookinfo 各 Pod 的重启情况"、"哪些 Pod 有异常重启" |
| PromQL 模板 | `sum(kube_pod_container_status_restarts_total{namespace="..."}) by (pod)` |
| 分级警告 | 🟢 0 次正常 / 🟡 1-2 次关注 / 🔴 ≥3 次异常 |
| 数据依赖 | 需要 kube-state-metrics 组件部署 |

#### 4.3.5 网络 I/O 监控

**工具名称**：`query_pod_network_io`

| 属性 | 内容 |
|------|------|
| 典型提问 | "查看 bookinfo 各服务的网络流量"、"productpage 的网络 I/O" |
| 查询 | 同时查询接收速率 `rate(container_network_receive_bytes_total[5m])` 和发送速率 `rate(container_network_transmit_bytes_total[5m])` |
| 输出格式 | 表格形式，列对齐显示 Pod、接收速率、发送速率 |

#### 4.3.6 一站式健康报告

**工具名称**：`query_namespace_health_report`

| 属性 | 内容 |
|------|------|
| 典型提问 | "bookinfo 命名空间整体健康状态如何"、"查看 bookinfo 的运维报告" |
| 能力 | **一站式聚合**：同时查询 CPU、内存、重启次数并做综合评估 |
| 评估维度 | CPU 异常偏高检测 / 内存超 1 GiB 预警 / 频繁重启检测 |

该工具是**多工具协同**的典范——它在内部依次调用多个 PromQL 查询，然后将结果汇总为一份结构化的健康报告，并给出综合评估结论。

#### 4.3.7 Prometheus 健康检查

**工具名称**：`prometheus_health_check`

| 属性 | 内容 |
|------|------|
| 典型提问 | "Prometheus 是否正常运行"、"检查监控系统状态" |
| 检查项 | `/-/ready` 端点检查 / 版本信息获取 / 活跃 Targets 统计 |

#### 4.3.8 Jaeger 调用链查询

**工具名称**：`jaeger_query_tool`

### 4.4 系统提示词工程

为了让 LLM 在监控场景下表现精准，设计了专门的 **`MONITORING_SYSTEM_PROMPT`**，包含以下关键设计：

#### 4.4.1 角色定位
```text
You are an intelligent monitoring assistant with expert-level knowledge of
Prometheus, PromQL, and Kubernetes observability.
```

#### 4.4.2 问题到工具的映射表
提示词中内置了**常见问题到工具名称的映射**，引导 LLM 理解用户意图与工具之间的对应关系：
```text
- "CPU usage/占用/使用率" → query_pod_cpu_usage
- "内存占用/使用量/Memory" → query_pod_memory_usage
- "重启/崩溃/异常" → query_pod_restart_count
- "健康/状态总览/报告" → query_namespace_health_report
- "趋势/变化/曲线图/图表" → query_cpu_trend or query_memory_trend
- "网络/流量/IO" → query_pod_network_io
- "Prometheus/监控系统状态" → prometheus_health_check
```

#### 4.4.3 交互模式约束
提示词规范了智能体的行为流程：
1. **理解意图** → 识别用户关心的指标类型
2. **选择工具** → 基于问题-工具映射选取最合适的工具
3. **执行查询** → 使用正确的参数调用工具
4. **分析结果** → 识别异常、计算占比、判断趋势
5. **生成报告** → 以结构化格式呈现发现并给出可操作建议

#### 4.4.4 输出风格约束
- 使用 Emoji 标记不同部分（🔥 CPU、💾 内存、🔄 重启等）
- 按照"数据→分析→建议"的三段式结构组织回答
- 对异常情况给出明确的警告和建议

### 4.7 技术要点总结

1. **工具描述工程**：每个工具的 Description 中嵌入了 2-3 个典型提问示例，这是引导 LLM 正确选择工具的关键——LLM 在遇到类似表述时会优先匹配描述中提到的工具。

2. **PromQL 选择策略**：不同指标使用最合适的 PromQL 变体——CPU 用 `rate(container_cpu_usage_seconds_total[5m])` 计算 5 分钟平均速率，内存用 `container_memory_working_set_bytes` 获取实际物理内存。

3. **数据后处理与分析**：工具不仅返回原始数据，还自动完成排序、格式转换（bytes→MiB）、异常检测（超过 3 倍平均值标记为异常）、趋势判断（时序数据分段比较）。

4. **多工具协同**：智能体可以自主组合多个工具完成复杂任务——先执行 CPU 查询，再执行内存查询，最后汇总为综合报告，无需人工干预。

5. **可视化集成**：matplotlib 趋势图使智能体不仅能"说"数据，还能"画"数据，大幅提升了运维人员对时序趋势的感知效率。

6. **错误处理链**：当 Prometheus 查询失败时，工具会返回结构化的错误提示并建议调用 `failure_diagnosis_tool`，形成完整的诊断闭环。

## 五、故障注入与智能诊断闭环（场景二）

### 5.1 场景描述

实验二展示了如何通过 Chaos Mesh 向 bookinfo 微服务系统注入网络分区、CPU 压力等故障，并通过 Jaeger 和 Prometheus 观察系统变化。本场景构建一个能够**自主完成故障诊断闭环**的智能体——当用户以自然语言描述异常现象时，智能体主动调用多种工具收集诊断依据，综合分析后输出根因定位与修复建议。

### 5.2 多工具协同诊断流程

```
用户描述异常
    ↓
Step 1: 查询 Prometheus 指标（CPU/内存/错误率）
Step 2: 查询 Jaeger 调用链（定位延迟瓶颈）
Step 3: 查看 Pod 日志（寻找错误堆栈）
Step 4: 检查 K8s 事件和 Chaos Mesh 配置
    ↓
综合多源数据 → 输出根因定位与修复建议
```

### 5.3 典型交互

```
用户：productpage 访问变慢了，帮我排查一下。

智能体：正在排查，请稍候……
  （调用工具：查询 productpage 的 CPU/内存指标）
  → 发现 CPU 使用率正常，内存无异常。
  （调用工具：查询近期 Jaeger Trace）
  → 发现 productpage → reviews 的调用耗时异常，平均延迟达 3.2s。
  （调用工具：查询 reviews 相关的 Pod 日志）
  → 日志中未见明显错误。
  
  诊断结论：问题可能源于 reviews → ratings 之间存在网络延迟。
  建议检查是否有活跃的 NetworkChaos 配置，
  或执行 kubectl describe networkChaos -n bookinfo 确认故障注入状态。
```

### 5.4 技术要点

本场景的核心难点在于**多工具的协同调用**：

- 智能体需在一次对话中依次调用 Prometheus、Jaeger、kubectl 等多种工具
- 工具粒度设计至关重要：太细则调用步骤繁琐，太粗则缺乏灵活性
- 通过提示词引导智能体在信息不足时主动补充查询，而非草率下结论
- 在与示例项目结合时，`failure_diagnosis_tool` 是实现诊断闭环的关键组件

## 七、实验思考与总结

### 7.1 ReAct 循环的工程价值

通过本实验，我们深刻体会到 ReAct（Reasoning + Action）循环在智能体中的核心作用。在测试运行中，智能体面对 Dockerfile 构建错误（`npm ci` 缺少 lockfile、多阶段构建目录为空）时，能够：

1. **思考（Thought）**：分析错误日志，定位根因
2. **行动（Action）**：修改 Dockerfile，调整构建策略
3. **观察（Observation）**：验证新方案是否成功
4. **迭代**：直至任务完成

这与手册中描述的"动态循环"完全一致，展示了智能体相比于传统"链（Chains）"的优势——执行路径由 LLM 自主决定，而非预先固定。

### 7.2 工具粒度设计经验

| 粒度 | 优点 | 缺点 |
|------|------|------|
| **过细** | 灵活组合 | 调用步骤多，易超迭代限制 |
| **适中** ✅ | LLM 可有效路由 | 需精心设计描述 |
| **过粗** | 调用少 | 缺乏灵活性，难以复用 |

本实验采用了适中粒度，每个工具对应一个原子操作（读文件、写文件、构建镜像等），通过 LLM 的推理能力组合为复杂工作流。

### 7.3 从 Docker 到 K8s 的扩展总结

| 能力维度 | Docker 阶段（手册示例） | K8s 扩展（本实验） |
|----------|----------------------|-------------------|
| **配置生成** | Dockerfile | Deployment + Service YAML |
| **部署执行** | `docker run` | `kubectl apply` |
| **状态验证** | 容器运行状态 | Pod Ready 状态轮询 |
| **资源管理** | 容器生命周期 | 命名空间 + 资源编排 |
| **网络暴露** | 端口映射 | Service（ClusterIP/NodePort/LoadBalancer） |

### 7.4 四个场景对比

| 场景 | 核心能力 | 难度 | 工程价值 |
|:----|:---------|:----:|:--------:|
| **场景一**：智能监控问答 | PromQL → 自然语言 + 可视化 | ⭐⭐⭐ | 降低监控使用门槛，提升数据可读性 |
| **场景二**：故障诊断闭环 | 多工具协同推理 | ⭐⭐⭐ | 提升排障效率 |
| **场景三**：K8s 部署流水线 | 全自动 CI/CD | ⭐⭐⭐ | **最高，可直接用于生产** |
| **场景四**：平台集成 | 接口标准化 | ⭐⭐ | 低代码快速构建 |

### 7.5 场景一专项总结

本实验重点实现了**场景一（智能监控问答助手）**，在以下方面取得了显著成果：

1. **工具层次化设计**：从底层 PromQL 查询到上层健康报告，形成了 4 层工具架构，兼顾灵活性和易用性。
2. **自然语言理解**：通过精心设计的工具名称和描述（含典型提问示例），LLM 能够准确理解"CPU 趋势"、"内存排名"、"重启情况"等自然语言表述并映射到正确的工具。
3. **结构化报告生成**：每个工具不仅返回原始数据，还自动完成排序、格式化、异常检测和趋势分析，最终输出运维人员可直接阅读的分析报告。
4. **可视化能力**：趋势查询工具集成 matplotlib，可自动生成带时间轴的专业折线图，使时序数据的变化趋势一目了然。
5. **一站式健康报告**：`query_namespace_health_report` 工具组合多个 PromQL 查询，一次调用即可获得命名空间的完整健康评估。

### 7.6 改进方向

1. **多步骤计划**：引入 LangGraph，让智能体先规划再执行，提高稳定性
2. **状态跟踪与自恢复**：记录每一步状态，出错时回滚重试而非从头开始
3. **告警集成**：将 Prometheus Alertmanager 告警接入智能体，实现告警自动响应
4. **RAG 知识库**：将 K8s 文档、Istio 手册等作为外部知识注入
5. **多智能体协作**：拆分为 Plan Agent、Execute Agent、Diagnose Agent
6. **时序异常检测**：引入统计方法（如 3-sigma、移动平均）自动识别指标异常
