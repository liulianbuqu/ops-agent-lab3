## 一、实验概述

### 1.1 实验目标

本实验选择**场景一：智能监控问答助手**作为实现目标。核心任务是将 Prometheus 查询能力封装为具有清晰语义的智能体工具，使运维人员可以通过**自然语言**向智能体提问，由智能体自主选择合适的 PromQL 查询语句、调用 Prometheus API、汇总数据并生成可读的分析报告。

具体目标：

1. 掌握基于 LangChain 框架的 ReAct 智能体构建方法
2. 系统性地将 Prometheus 查询接口封装为多个具有清晰语义的工具
3. 通过精心设计的工具描述（Description）引导 LLM 在不同问题下选择正确的工具组合
4. 结合 matplotlib 实现趋势可视化，让智能体在回答数据趋势问题时自动生成折线图
5. 实现一站式命名空间健康报告聚合能力

### 1.2 设计思路

本场景的核心设计哲学是：**"工具即语义"**——将 Prometheus 的查询接口拆分为多个粒度适中的原子工具，每个工具通过名称、描述和参数与特定的监控语义绑定。工具之间形成层次化体系：

```
┌────────────────────────────────────────────────────────┐
│                    MonitorAgent                         │
│          (LangChain ReAct + DeepSeek Chat)              │
├────────────────────────────────────────────────────────┤
│                                                        │
│  工具层次：                                              │
│  ┌──────────────────────────────────────────────────┐  │
│  │ 第4层：聚合工具                                    │  │
│  │  query_namespace_health_report                   │  │
│  │  └→ 内部调用多个 PromQL，聚合为综合报告              │  │
│  ├──────────────────────────────────────────────────┤  │
│  │ 第3层：趋势工具（含可视化）                          │  │
│  │  query_cpu_trend / query_memory_trend             │  │
│  │  └→ 范围查询 + matplotlib 折线图                   │  │
│  ├──────────────────────────────────────────────────┤  │
│  │ 第2层：领域查询工具                                │  │
│  │  query_pod_cpu_usage / query_pod_memory_usage    │  │
│  │  query_pod_restart_count / query_pod_network_io  │  │
│  │  └→ 封装特定指标的 PromQL + 数据后处理              │  │
│  ├──────────────────────────────────────────────────┤  │
│  │ 第1层：底层 PromQL 工具                            │  │
│  │  prometheus_query_tool (即时/范围查询)              │  │
│  │  prometheus_health_check                         │  │
│  │  └→ 直接与 Prometheus API 交互                     │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────┘
```

## 三、智能监控问答助手的实现（场景一）

### 3.1 监控工具集详细设计

本场景实现了 **11 个监控专用工具**，覆盖 CPU、内存、网络、重启、趋势、健康检查和分布式追踪。

#### 3.1.1 底层通用查询工具

**`prometheus_query_tool`** — 执行原始 PromQL 查询

```python
@tool("prometheus_query_tool")
def prometheus_query_tool(
    promql: str,
    prometheus_url: str = "http://localhost:9090",
    use_range: bool = False,
    duration_minutes: int = 5,
    step_seconds: int = 15
) -> str:
```

- 支持即时查询（instant query）和范围查询（range query）两种模式
- 即时查询通过 `prom.custom_query()` 实现，返回当前值
- 范围查询通过 `prom.custom_query_range()` 实现，返回时序数据点
- 当专用工具无法覆盖的高级查询场景时使用

#### 3.1.2 CPU 监控工具

**`query_pod_cpu_usage`** — 查询 Pod CPU 使用率

| 属性 | 内容 |
|------|------|
| 工具名称 | `query_pod_cpu_usage` |
| 典型提问 | "查看 bookinfo 各服务的 CPU 使用情况"、"productpage 的 CPU 占用" |
| PromQL | `sum(rate(container_cpu_usage_seconds_total{namespace="..."}[5m])) by (pod)` |
| 数据分析 | 按 CPU 使用量降序排列，自动格式化（cores/m），计算占比 |
| 异常检测 | 使用 3 倍平均值法：`value > avg * 3` 标记为异常 |

**PromQL 说明**：
- `container_cpu_usage_seconds_total` 是容器累计 CPU 使用时间（秒）
- 使用 `rate()[5m]` 计算 5 分钟平均增长率，消除瞬时波动
- `by (pod)` 按 Pod 维度聚合，支持多容器 Pod

**输出示例：**
```
🔥 CPU 使用率报告 — 命名空间: bookinfo
==================================================
  📦 productpage-v1-xxxxx
     CPU: 0.035 cores
  📦 reviews-v1-xxxxx
     CPU: 0.012 cores
  ...
==================================================
  📈 总计 CPU 使用: 0.089 cores
  💡 分析: productpage 的 CPU 占用最高，占整体的 39.3%。
```

#### 3.1.3 内存监控工具

**`query_pod_memory_usage`** — 查询 Pod 内存使用量

| 属性 | 内容 |
|------|------|
| 工具名称 | `query_pod_memory_usage` |
| 典型提问 | "bookinfo 各服务的内存占用情况"、"reviews-v2 内存用了多少" |
| PromQL | `sum(container_memory_working_set_bytes{...}) by (pod)` |
| 数据分析 | 自动转换为 MiB 格式，检测超过 1 GiB 的高内存预警 |

> **设计要点**：使用 `container_memory_working_set_bytes` 而非 `container_memory_usage_bytes`。Working Set 更能反映容器实际使用的物理内存，排除了缓存的影响，更接近容器 OOM 的判断标准。

#### 3.1.4 重启次数监控

**`query_pod_restart_count`** — 查询 Pod 重启次数

| 属性 | 内容 |
|------|------|
| 工具名称 | `query_pod_restart_count` |
| 典型提问 | "查看 bookinfo 各 Pod 的重启情况"、"哪些 Pod 有异常重启" |
| PromQL | `sum(kube_pod_container_status_restarts_total{...}) by (pod)` |
| 分级告警 | 🟢 0 次正常 / 🟡 1-2 次关注 / 🔴 ≥3 次异常 |

> **依赖说明**：此查询依赖 kube-state-metrics 组件提供的 `kube_pod_container_status_restarts_total` 指标。如果未部署 kube-state-metrics，工具会返回提示信息。

#### 3.1.5 网络 I/O 监控

**`query_pod_network_io`** — 查询网络接收/发送速率

同时查询两个 PromQL：
- 接收速率：`rate(container_network_receive_bytes_total[5m])`
- 发送速率：`rate(container_network_transmit_bytes_total[5m])`

输出以表格形式展示，列对齐显示 Pod 名称、接收速率和发送速率。

#### 3.1.6 趋势分析与可视化

**`query_cpu_trend`** — CPU 时序趋势查询

**`query_memory_trend`** — 内存时序趋势查询

这两个工具的核心实现：

```python
# 范围查询获取时序数据
result = prom.custom_query_range(
    query=promql,
    start_time=start_time,
    end_time=end_time,
    step=step_seconds
)

# 分段比较自动判断趋势
if len(vals) >= 5:
    first_half = sum(vals[:len(vals)//2]) / max(len(vals)//2, 1)
    second_half = sum(vals[len(vals)//2:]) / max(len(vals)-len(vals)//2, 1)
    change = (second_half - first_half) / max(first_half, 0.0001) * 100
    if change > 20:     → "显著上升"
    elif change < -20:  → "显著下降"
    else:               → "平稳"
```

当 `save_chart=True` 时，自动调用 matplotlib 生成折线图：

```python

```

#### 3.1.7 一站式健康报告

**`query_namespace_health_report`** — 聚合多项指标生成综合报告

该工具在内部依次执行以下 PromQL 查询：
1. CPU 使用率：`rate(container_cpu_usage_seconds_total[5m])`
2. 内存使用量：`container_memory_working_set_bytes`
3. 重启次数：`kube_pod_container_status_restarts_total`

然后进行三项异常检测：
- CPU 异常偏高（超过平均值的 3 倍）
- 内存超过 1 GiB
- 频繁重启（≥3 次）

最终输出结构化的健康报告及综合评估结论。

#### 3.1.8 Prometheus 健康检查

**`prometheus_health_check`** — 检查 Prometheus 服务状态

通过三个端点验证：
1. `/-/ready` — 检查 Prometheus 就绪状态
2. `/api/v1/status/buildinfo` — 获取版本信息
3. `/api/v1/targets` — 获取活跃 Targets 数量

#### 3.1.9 Jaeger 调用链查询

**`jaeger_query_tool`** — 查询分布式追踪数据

通过调用 Jaeger HTTP API 实现：
- `GET /api/services` — 列出所有已注册服务
- `GET /api/traces?service=...&limit=...` — 查询特定服务的追踪

### 3.2 MonitorAgent 构建

**文件：`core/agent_builder.py`**

```python
class MonitorAgent:
    """
    专为"智能监控问答助手"设计的 ReAct Agent
    """

    def _init_tools(self):
        return [
            # 监控核心工具（9个）
            prometheus_query_tool,          # 通用 PromQL 查询
            query_pod_cpu_usage,            # CPU 使用率排名
            query_pod_memory_usage,         # 内存使用排名
            query_pod_restart_count,        # 重启次数检测
            query_namespace_health_report,  # 健康报告
            query_cpu_trend,                # CPU 趋势 + 折线图
            query_memory_trend,             # 内存趋势 + 折线图
            query_pod_network_io,           # 网络 I/O
            prometheus_health_check,        # Prometheus 健康检查
            # 追踪工具
            jaeger_query_tool,
            # 诊断工具
            failure_diagnosis_tool,
        ]

    def _build_agent(self):
        return initialize_agent(
            tools=self.tools,
            llm=self.llm,
            agent=AgentType.STRUCTURED_CHAT_ZERO_SHOT_REACT_DESCRIPTION,
            verbose=self.verbose,
            max_iterations=30,
            handle_parsing_errors=True,
            agent_kwargs={
                "system_message": MONITORING_SYSTEM_PROMPT,
            },
        )
```

### 3.3 数据后处理与智能分析

每个工具不仅返回原始 Prometheus 查询结果，还自动完成以下数据处理：

| 处理步骤 | 方法 | 示例 |
|---------|------|------|
| **格式转换** | bytes → MiB 格式化 | `303187456 → 303.2 MiB` |
| **降序排序** | `sorted(parsed, key=lambda x: x["value"], reverse=True)` | 最高使用量排首位 |
| **占比计算** | `value / total * 100` | "productpage 占整体的 36.1%" |
| **异常检测** | CPU: `value > avg*3` / 内存: `>1GiB` / 重启: `≥3次` | "⚠️ 建议关注" |
| **趋势判断** | 时序数据分段均值比较 | "显著上升(+25%) / 平稳(+3%)" |
| **分级告警** | 🟢正常 / 🟡关注 / 🔴异常 | 重启次数分级图标 |

## 五、实验思考与总结

### 5.1 ReAct 循环的工程价值

通过本实验，我们深刻体会到 ReAct（Reasoning + Action）循环在智能体中的核心作用。在测试过程中，智能体面对 Prometheus 连接失败的情况时，能够：

1. **思考（Thought）**：分析错误日志，判断需要检查 Prometheus 状态
2. **行动（Action）**：调用 `prometheus_health_check` 验证
3. **观察（Observation）**：获取服务不可用的结果
4. **迭代**：调用 `failure_diagnosis_tool` 进一步诊断

这与手册中描述的"动态循环"完全一致——执行路径由 LLM 自主决定，而非预先固定。

### 5.2 工具粒度设计的经验

| 粒度 | 优点 | 缺点 | 本场景策略 |
|------|------|------|-----------|
| **过细** | 灵活组合 | 调用步骤多，易超迭代限制 | ❌ 避免 |
| **适中** ✅ | LLM 可有效路由 | 需精心设计描述 | ✅ 采用 |
| **过粗** | 调用少 | 缺乏灵活性 | ❌ 避免 |

本场景采用了**适中粒度**：每个工具对应一个具体的监控语义（CPU 查询、内存查询、重启查询等），通过 LLM 的推理能力组合为复杂监控任务。

### 5.3 提示词工程的关键作用

实验证明，工具 Description 的设计对 LLM 的工具选择准确率有决定性影响。关键经验：

1. **嵌入典型提问**：每个工具的 Description 中包含了 2-3 个典型提问示例，LLM 在遇到类似表述时会优先匹配
2. **明确输入输出**：清晰说明参数含义和返回数据格式
3. **中英双语覆盖**：Description 和提示词同时覆盖中英文问法

### 5.4 技术要点总结

1. **PromQL 选择策略**：不同指标使用最合适的 PromQL 变体——CPU 用 `rate(container_cpu_usage_seconds_total[5m])` 计算 5 分钟平均速率，内存用 `container_memory_working_set_bytes` 获取实际物理内存

2. **数据后处理**：工具不仅返回原始数据，还自动完成排序、格式转换（bytes→MiB）、异常检测（3σ 法）、趋势判断（时序数据分段比较）

3. **可视化集成**：matplotlib 趋势图使智能体不仅能"说"数据，还能"画"数据，大幅提升了运维人员对时序趋势的感知效率

4. **错误处理链**：当 Prometheus 查询失败时，工具会返回结构化的错误提示并建议调用 `failure_diagnosis_tool`，形成完整的诊断闭环

### 5.5 改进方向

1. **多步骤计划**：引入 LangGraph，让智能体先规划再执行，提高复杂任务的稳定性
2. **异常检测增强**：引入统计方法（如 3-sigma、移动平均）自动识别指标异常
3. **告警集成**：接入 Prometheus Alertmanager，实现告警的自动响应与处理
4. **时序预测**：结合 Prophet 或 LSTM 实现对指标的短期趋势预测
